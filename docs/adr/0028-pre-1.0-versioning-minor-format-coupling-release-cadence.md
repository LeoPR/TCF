# 0028 — Versionamento pré-1.0: acoplamento minor↔formato + cadência de release

**Status**: accepted (2026-07-09)
**Date**: 2026-06-24 (accepted 2026-07-09)
**Deciders**: project owner

> **accepted 2026-07-09** junto com o [ADR-0032](0032-tcf8-default-format.md) (#TCF.8 default). Reconciliação
> da pendência registrada: o magic `#TCF.8` shipou opt-in sob `0.7.1` (scaffold byte-neutro, default-off) —
> NÃO violou o coupling porque não era o formato default; com o ADR-0032 o `#TCF.8` VIRA default → o pacote
> vai a `0.8.0` (regra 1). O ciclo `0.7.2` (lazy+poda) é **absorvido** no `0.8.0` (sem release intermediário,
> decisão do owner 2026-07-09). PyPI publica só quando completo+estável.
**Tags**: versioning, policy, pre-1.0, release-cadence, format, pypi
**Refines**: [ADR-0024](0024-pre-1.0-versioning-git-as-compat.md) (NÃO revoga — refina a aplicação do coupling minor↔formato)
**Relaciona**: [ADR-0017](0017-format-spec-v1-frozen.md) (freeze v1.0 já superseded por 0024), [ADR-0027](0027-nature-mark-header-self-describing.md) (exemplo correto de bump de minor por formato: #TCF.8 = 0.8)

## Context and Problem Statement

[ADR-0024](0024-pre-1.0-versioning-git-as-compat.md) (2026-06-14) fixou o projeto em pré-1.0, com o
pacote em `0.x` "alinhado ao `#TCF.7`" e o minor do formato `#TCF.N` como iteração de
desenvolvimento. A INTENÇÃO — "o minor acompanha o formato" — está correta. Mas a 0024 fundiu MINOR e
FORMATO sem nomear um terceiro eixo: o **contador de RELEASE dentro de um mesmo formato**.

A omissão produziu inflação de versão. Lendo a 0024 literalmente, cada estado vira `0.N.0` e qualquer
entrega nova pediria bump de minor. Isso levou o ciclo do lazy (`tcf.view`) + poda de legado — que NÃO
mudam o formato (segue `#TCF.7`) — a ser rotulado `0.8.0` e "plano 0.8" em STATUS/ROADMAP/notas/tickets.
O caso exato: o doc dos 3 eixos (`plano-poda-legado-pre-0.7-2026-06-24.md`) DEFINE a regra "não
atropelar o sincronismo formato↔desenvolvimento" mas a própria tabela escreve `0.7.1 → 0.8.0` para uma
poda que não toca o formato. Em paralelo, as notas atribuíram o cross-dict (`#TCF.8`) a `0.9`,
contradizendo "minor acompanha o formato".

Este ADR refina a 0024: separa explicitamente o eixo RELEASE e ancora a regra de bump.

## Decision Outcome

### Três eixos de versão (distintos, ortogonais)

- **(A) Versão de FORMATO** — shebang `#TCF.N`. Contrato on-disk. Só muda com mudança de formato.
  Hoje: `#TCF.7` (default), `#TCF.6` (legado, lido).
- **(B) Geração do ENCODER** — marco interno do algoritmo (M8A → M9 → M10). Bytes diferentes DENTRO da
  mesma família de formato (M9 é subset de M10, mesmo decoder). NÃO é versão pública; vira nota
  histórica. Nunca aparece no número do pacote.
- **(C) Versão do PACOTE (PyPI)** — pré-1.0 = `0.<formato>.<release>`. O **minor** = número do formato
  (`0.N` ↔ `#TCF.N`). O **release** (patch) = contador de entregas DENTRO daquele formato.

### Regra de bump (pré-1.0)

1. Mudança de FORMATO (`#TCF.N` → `#TCF.N+1`) **move o minor**: `0.(N+1).0`.
2. Acessório / fix / poda / perf com **formato inalterado** move só o **release/patch**:
   `0.N.x → 0.N.(x+1)`.
3. O número MAJOR não se move pré-1.0 (fica `0`). O coupling antigo "`#TCF.N` → pacote `N+1.0.0`"
   (ADR-0018, ADR-0022) fica obsoleto; substituído por "`#TCF.N` → `0.N.0`".

Exemplos canônicos:
- **lazy + poda de legado** = formato `#TCF.7` inalterado ⇒ **release `0.7.2`** (NÃO `0.8.0`). O que se
  chamava "plano 0.8" é o ciclo `0.7.2`.
- **cross-dict (H-GDICT)** = weld de `#TCF.8` ⇒ **minor `0.8.0`**. `0.8.0` fica RESERVADO pro `#TCF.8`.

### Cadência PyPI (segura)

- Versão do repo permanece `0.7.1` (última release publicada). Bump pra `0.7.2` só quando o release do
  lazy for deliberado.
- PyPI segura: publish só com go explícito do owner (Trusted Publishing via `release.yml`). A tag do
  release do lazy é `v0.7.2`; `v0.8.0` reservada pro ciclo `#TCF.8`.

### Transição 1.0

- `1.0` quando o formato final congelar (`#TCF.8`/`.9`). A partir do 1.0, semver estrito (X.Y.Z,
  breaking = major) passa a valer — compat importa (acervo real, usuários). Pré-1.0 NÃO usa a tabela
  semver `1.0.x/1.x.0/2.0.0`.

### Invariantes preservados (NÃO revogados)

- **git-as-compat** (ADR-0024): `git checkout` reproduz/decoda eras antigas. Continua sendo o
  mecanismo de compatibilidade pré-1.0.
- **Baselines de byte re-pináveis** (ADR-0024): poda/perf que não mudam formato re-pinam baseline e
  movem o RELEASE, não o minor.
- **Reenquadramento "v2.0" = "0.7 pré-1.0"** (ADR-0024): estendido aqui aos welds 0025/0026 (além de
  0018/0022/0023), que também se rotulam "v2.0" mas são do formato `#TCF.7` — foram entregas no eixo
  RELEASE.

## Consequences

- **Positive**: fecha a inflação `0.8.0` para ciclos sem mudança de formato; o número do pacote passa
  a sinalizar inequivocamente o formato (minor) e a cadência de entrega (release); o `#TCF.8` ganha um
  destino único e estável (`0.8.0`, não `0.9`).
- **Negative/limites**: artefatos datados já escritos com a nomenclatura "0.8"/"plano 0.8"
  (checkpoints, entries de ticket, filename `v08-plano-etapas.md`) ficam stale — tratados por nota de
  retificação, não reescrita (são registros datados).
- **Neutral**: o eixo (B) encoder M9/M10 passa a ser nomeado como eixo de versão interno; não altera
  nenhum número publicado.

## Relação com ADRs anteriores

- **ADR-0024** — REFINADO (não revogado). A cláusula "minor acompanha o formato" permanece; este ADR
  adiciona o eixo RELEASE e a regra de bump que evita `0.8.0` com formato `#TCF.7`. git-as-compat e
  baselines re-pináveis seguem invariantes.
- **ADR-0017** — já superseded-parcial (freeze v1.0) pela 0024. Este ADR não toca o corpo do 0017;
  apenas reafirma que a tabela semver `1.0.x/1.x.0/2.0.0` do 0017 não vale pré-1.0.
- **ADR-0027** — citado como aplicação CORRETA: `#TCF.7 → #TCF.8 = 0.8` (mudança de formato move o
  minor).
