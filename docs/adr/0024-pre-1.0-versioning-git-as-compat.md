# 0024 — Versionamento pré-1.0: minors de desenvolvimento, git como compatibilidade

**Status**: accepted
**Date**: 2026-06-14
**Deciders**: project owner
**Tags**: versioning, policy, pre-1.0, compatibility, git
**Supersedes**: a parte "freeze v1.0" de [ADR-0017](0017-format-spec-v1-frozen.md)
**Refined-by**: [ADR-0028](0028-pre-1.0-versioning-minor-format-coupling-release-cadence.md) — separa o
eixo RELEASE (`0.N.x`) do MINOR (=formato); a regra "minor acompanha o formato" permanece, mas entrega
sem mudanca de formato move o PATCH, nao o minor (evita inflar `0.8.0` com formato `#TCF.7`).
git-as-compat e baselines re-pinaveis seguem invariantes.

## Context and Problem Statement

[ADR-0017](0017-format-spec-v1-frozen.md) declarou "format spec v1.0 frozen" e o
pacote foi rotulado `1.0.0`. Mas o formato segue em **remodelagem ativa** — so'
em 2026-06-14 o `#TCF.7` mudou duas vezes (weld de min_header + revisao que
dropou o prefixo do meta). Chamar de "congelado v1.0" enquanto se remodela o
formato e' incoerente. O projeto na verdade esta' **pré-1.0**.

O owner reenquadrou (2026-06-14): as versoes feitas ate' agora (#TCF.4/.5/.6/.7)
sao **evolucao em direcao a um 1.0 solido**, nao releases estaveis nem um v2.0.

## Decision Outcome

**O projeto e' PRÉ-1.0.** Política:

1. **Pacote em `0.x`** (era `1.0.0`, rotulo prematuro -> `0.7.0`, alinhado ao
   `#TCF.7`). Minors do formato `#TCF.N` = **iteracoes de desenvolvimento**, que
   servem de **marcador de comparacao/controle de experimento**, NAO de contrato
   de compatibilidade.

2. **Sem compat rigida entre minors pré-1.0.** O encoder escreve o formato mais
   novo; o decoder **round-trippa o atual**. Ler formatos ANTIGOS e' conveniencia
   OPCIONAL (no maximo a versao imediatamente anterior, "pra durar mais"), nao
   obrigacao.

3. **Git e' o mecanismo de compatibilidade/reprodutibilidade.** Pra reproduzir
   ou decodar um experimento de uma era antiga: `git checkout` do commit daquela
   versao e roda aquele codigo. O numero de versao diz QUAL git buscar. Os `.tcf`
   sao artefatos experimentais regeneraveis, nao um acervo de producao — entao
   git basta.

4. **Baselines de byte (D1-D9=1523B, D17a=322B, snapshots) = guardas de regressao
   ACIDENTAL, re-pinaveis em mudanca INTENCIONAL de formato.** Nao sao contrato
   eterno. (Mantem a protecao que pegou o bug do retail em 2026-06-13; muda so' a
   promessa de "eterno" pra "baseline atual".)

5. **v1.0 = release solido FUTURO** (quando o formato estabilizar). A PARTIR do
   1.0 a compatibilidade passa a importar (acervo real, usuarios). Ao fechar o
   1.0, pode-se **limpar** o tratamento das versoes intermediarias (0.4/0.5/0.6)
   — inclusive o decode delas, ja' que git guarda a historia.

6. **v2.0 = depois**, com fundamentacao maior (ideias em [ADR-0018](0018-v2-format-roadmap.md)).
   **CUIDADO COM A NOMENCLATURA**: `#TCF.7` = **"0.7" pré-1.0**, NAO v2.0. Os
   ADRs [0022](0022-v2a-fallback-identity-weld.md)/[0023](0023-v2-minimal-header-weld.md)
   que dizem "abre v2.0" devem ser lidos como "0.7 dentro da linha pré-1.0".

## O que continua de ADR-0017

A distincao **format version (`#TCF.N`) vs library version (semver)** e a
disciplina de versionar o formato continuam. So' cai o "frozen forever em v1.0":
`#TCF.6` deixa de ser congelado-eterno e vira o **snapshot 0.6** (uma era de dev).

## Consequences

- **Positive**: honestidade (nao "congelado" enquanto muda); liberdade de refinar
  pré-1.0; codigo vivo sem cruft de compat (git carrega historia); caminho claro
  pra um 1.0 limpo.
- **Negative/limites**: `.tcf` gerados hoje podem nao decodar com codigo futuro
  (aceito pré-1.0 — sao regeneraveis; git reproduz). Ir "pra tras" no semver
  (1.0.0 -> 0.7.0) e' incomum, mas o 1.0.0 nunca foi release publicado — e'
  correcao de rotulo.
- **Neutral / pendente (decisao tecnica separada)**: tornar `#TCF.7` o DEFAULT
  do encode (hoje opt-in via `fallback`/`min_header`) — re-pinaria D17a; segue
  como proximo passo quando o owner mandar.
