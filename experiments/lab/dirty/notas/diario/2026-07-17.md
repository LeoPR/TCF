## Madrugada (2) — perfil "json-like": reavaliação medida + condições para parâmetro

Owner: *"warning para comportamento similar ao json para essas condições... um default com um
conjunto de comportamentos json-like é útil. Reavalie e emita as condições para ter parâmetro."*

Registro: `notas/perfil-json-like-condicoes-parametro.md`. Reavaliei MEDINDO 10 bordas
(json × `.8H`) antes de opinar — e a medição mudou a resposta.

**"json-like" não é um eixo — são 4 categorias**:
- **A · CAPACIDADE** (json faz, TCF não; 4): chave `""` (JSON válido, comum), `\n` em valor
  (ValueError CRU), chave com `\n`, ordem por-registro. Warning aqui = "avisei que não sei fazer"
  → o certo é IMPLEMENTAR. Nenhuma vira parâmetro.
- **B · POLÍTICA de adaptador** (1): chave duplicada em texto — única borda onde warning é certo,
  e só existe FORA do core (a info morre no dict).
- **C · DEFEITO do json** (5): NaN/`Infinity` são **inválidos por RFC 8259** e o Python emite por
  DEFAULT (`allow_nan=True`); NaN quebra RT (`nan != nan` — restrição inegociável do owner em
  H-HIER-SCALAR-01); `tuple`→`list` perde tipo; **chave não-str EMITE duplicata** (é o único
  caminho que fabrica o problema do §B!); lone surrogate faz RT mas não é UTF-8 transmissível.
  json-like aqui IMPORTA o bug — **TCF estrito é FEATURE**.
- **D · ALÉM do json**: representar em vez de tolerar (NaN/Inf tipados) → é FORMATO, não flag.
- ✅ **paridade já OK** (medida): `\t`, `\x00`, int gigante, `-0.0`, `0.1+0.2`.

**As 5 condições emitidas** (o teste pedido): C1 informação viva no ponto de decisão · C2 ambos
lossless ou perda declarada E observável (default nunca é o lossy) · C3 não muda semântica do
wire (se muda = FORMATO/discriminador, não flag) · C4 default decidível (doutrina ou medição) ·
C5 fail-loud continua alcançável. **Regra-mestra**: parâmetro é para POLÍTICA, nunca para
CAPACIDADE (→ implementar) nem para CORREÇÃO (→ o estrito é o certo; evoluir por representação).

**Teste aplicado: das 10 bordas, exatamente 1 vira parâmetro** — e mora no ADAPTADOR
(`strict` default = contrato S0 × `json_like` opt-in, com cada desvio reportado).

**CONDIÇÃO 0 BLOQUEANTE (medida)**: o `.8H` **não tem canal de efeito colateral** — `side_outputs`/
`SideOutputs` não aparecem em `hierarchical.py` (o flat tem). Hoje um warning no `.8H` não tem onde
morar. Precedente existe (`warnings.warn(UserWarning)` em `multi/core.py:327`, nome vazio→anônima),
mas `warnings` é global/dedupado — ruim em stream. SideOutputs é a ponte oficial e `anomaly_flags`
já está previsto (CLAUDE.md:240) → **H-HIER-SIDEOUTPUTS-01** é pré-requisito de qualquer política
com warning (e destrava profiler/schema-tool sobre hierarquia).

Hipóteses novas: H-PROFILE-JSONLIKE-01 · H-HIER-SIDEOUTPUTS-01. Tickets atualizados com as
lacunas/paridades medidas (JSON-PARITY) e os 3 erros crus (API-BOUNDARY-CONTRACTS). src/tcf INTACTO.

## Madrugada (3) — critério do FLUXO + escala de implementação por ROI

Owner fixou o eixo: implementar só o que o JSON consegue num ambiente JSON
(`dataset→encode→json→transmite→recebe→json→decode→dataset`; "basta o tcf ter comportamento
similar"), separar "problema do Python" de "problema do JSON" (*"no C, C++, Rust ou Java esse
problema existe? o que a norma dita?"*), e montar uma escala do mais barato ao completo.

**O critério, formalizado e MEDIDO** (lab `2026-07-17-0140`):
`∀D: J-RT-TX(D) ⟹ T-RT(D)` — com a etapa **TRANSMITE em bytes UTF-8**, não no str em memória
(sem ela o lone surrogate "passaria" pelo escape ASCII e mediríamos paridade que não existe no fio).

**PLACAR: PARIDADE=14 · LACUNA=3 · LACUNA-RAIZ=7 · TCF-ESTRITO=2 · AMBOS-RECUSAM=7.**
A superfície inteira de implementação = **3 lacunas de dataset** (chave `""`, `\n` em valor, chave
com `\n`) + **o eixo raiz (P4b)**. Todo o resto já é paridade ou é o json falhando/saindo da norma.
Pinado: `tests/test_json_flow_parity.py` (24 passed + 3 `xfail(strict)` — lacuna não fecha calada).
Bônus medido: **TCF ⊃ I-JSON** em inteiros > 2^53 (RFC 7493 §2.2 proíbe; nós fazemos RT).

**"Problema do Python ou do JSON?" — provado por COMPILAÇÃO** (`rustc 1.82`, o alvo do 1.0):
- chave int + chave str no mesmo mapa (o que **fabrica** a duplicata do `json.dumps`) → **erro
  E0308** em Rust ("expected `i32`, found `&str`") → **problema do Python** (tipagem dinâmica).
- lone surrogate → `"\u{D800}"` **não compila**; `char::from_u32(0xD800)` = `None`;
  `String::from_utf8([ED,A0,80])` = `Err` → **problema do Python**, inexprimível em Rust.
- NaN → `f64::NAN == f64::NAN` = `false` (o rustc **avisa**) → **problema do IEEE 754**, universal;
  só se resolve REPRESENTANDO (E7), nunca tolerando.
→ **2 dos 5 defeitos somem de graça no port Rust do 1.0**. A rigidez do TCF não é parochialismo
Python: é o que o sistema de tipos do Rust impõe sozinho.

**Escala** (`notas/escala-implementacao-paridade-json.md`), ordem recomendada:
**E0 critério ✅** → **E1 tipar 3 erros crus** (baixo, não toca wire) → **E3 SideOutputs no `.8H`**
(destrava o warning + profiler) → **E2 chave `""`** (hoje "nome vazio" é sentinela de corrupção →
estudo-primeiro) → **E5 P4b** (7 lacunas de uma vez, maior ROI) → avaliar E4 → **E6 `\n` em valor:
SEGURAR** (toca o **L1**, re-pina D1-D9/D17a/real-world — é o T-FMT-ESCAPE-COMBINATORIAL-STUDY) →
E7 além-do-json (formato/versão).

**Fronteira honesta**: com E1+E2+E3+E5 a frase certa é *"paridade com o fluxo JSON, exceto `\n` em
valor e `\n` em nome"* — nunca "qualquer JSON". Suíte: **796 passed, 2 skipped, 5 xfailed**.

## Madrugada (4) — correção do owner: DOIS CONTRATOS (o TCF lê dataset, não JSON) + doc oficial

Owner corrigiu o enquadramento: *"o tcf não é json... ele lê um dataset que PODE ser resultado de
um json... temos que seguir documentos oficiais, não podemos inventar à toa só lendo usando o
json.dump... 'limite da linguagem' não é argumento — se a documentação declara implementação
completa, o erro é nosso"*. Registro central: `notas/dataset-json-dois-contratos.md`.

**A arquitetura**: Contrato A (da lib: texto→dataset→texto; nossa responsabilidade é CONHECER onde
a doc declara que quebra) ⊥ Contrato B (do TCF: dataset→tcf→dataset sobre D_json). A∧B ⟹
transporte fecha ("caso encerrado"). O core NUNCA vê texto JSON; adaptadores são exemplos
modularizados fora do core.

**Doc oficial do CPython 3.13 (fetch, verbatim)**: a doc NÃO alega conformidade — declara *"This
module does not comply with the RFC in a strict fashion"* e lista os desvios (NaN/Inf aceitos e
emitidos; nomes repetidos last-wins) COM os interruptores de modo estrito (`allow_nan=False` =
*"in strict compliance with the JSON specification"*, `parse_constant`, `object_pairs_hook`) e o
aviso *"loads(dumps(x)) != x if x has non-string keys"*. **D_json agora é definida por DOCUMENTO**
(tabela oficial de conversão + RFC 8259 §2/§4/§6/§8.1 + I-JSON): dict[str,·]·list·str·int·float
finito·bool·None, raiz livre. Chave não-str/tuple/NaN/surrogate estão FORA da classe **pela doc**,
não por opinião — logo não são paridade.

**Pesquisa cross-language (workflow, doc oficial citada; Java+verify caíram por spend-limit)**:
ninguém além do CPython emite o token `NaN` (JS→null NORMATIVO sem toggle; serde_json→null
SILENCIOSO — medido em 1.0.150 instalado; Go→UnsupportedValueError fail-loud, único; nlohmann→null;
RapidJSON strict). As 3 famílias de duplicata da RFC 8259 §4 EXISTEM (last-wins py/js/serde; erro
serde-typed; RapidJSON PRESERVA). Go tem merge (≠ last-wins) p/ maps e pode EMITIR duplicata via
TextMarshaler keys. nlohmann REORDENA chaves alfabeticamente (parse→dump não é byte-estável).
**⚠ PORT RUST 1.0**: serde_json defaults = NaN→null calado + números grandes CORROMPIDOS calado
(precisa `arbitrary_precision`) — quebraria fail-loud e o TCF ⊃ I-JSON de ints > 2^53; registrado
na memória do 1.0 e no §4 da nota.

(sanidade da lib contra a PRÓPRIA doc, 5 testes), D_JSON (o critério, só onde vale: 14 paridades +
